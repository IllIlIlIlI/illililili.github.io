Basic config

Configuration for a basic HoneyTrap Server, with only one service.

This is the configuration used for the quick start. You can find it on the github repository of HoneyTrap-configs.

Each field is described here after.

Config-docker.toml

[listener]
type="socket"

[service.ssh-simulator]
type="ssh-simulator"
credentials=["root:root", "root:password"]

[[port]]
ports=["tcp/8022", "tcp/8023"]
services=["ssh-simulator"]

[service.telnet]
type="telnet"
image="/data/screenshot-20171030174349.png"
server-name="WS-3244"

[[port]]
port="tcp/5900"
services=["telnet"]

[channel.console]
type="console"

[[filter]]
channel=["console"]

[[logging]]
output = "stdout"
level = "debug"

- [listener]

[listener]
type="socket"

[listener] defines how HoneyTrap listens to the incoming traffic.
In this example, with type="socket", only some configured ports are listened on.
You can define only one listener.

More information about listener in concepts/listeners.
More information about socket in configuration/listeners/socket.
More configurations for listeners in configuration/listeners.

- [service.ssh-simulator]

With socket chosen as listener, we need to explicitly declare each service we want to use.

In this config file, for simplicity, we defined only one service, ssh-simulator.

[service.ssh-simulator]
type="ssh-simulator"
credentials=["root:root", "root:password"]
For each service:

Firstly, we need to define the name of the service.
The convention is [service.<you_choose_the_name>].
The name can be freely chosen and is referred by in the channel output.

Secondly, we need to define the type of the service.
The convention is type=<official_name_of_service>.

Finally, for each service, we can define some “arguments”. Each service has its own arguments. If you don’t define them in the config file, the default arguments will be used.

In this example, we define ["root:root","root:password"] as credentials. If we delete this line of the config file, the default behaviour of ssh-simulator will be to allow all credentials.

More information about services in concepts/services.
More configurations for services in configuration/services.
More information about ssh-simulator in configuration/services/ssh-simulator.

- [[port]]

Each service defined in the config file needs a [[port]] defined.

[[port]]
ports=["tcp/8022", "tcp/8023"]
services=["ssh-simulator"]

Firstly, we need to define to which ports we want to link to the service.
The convention is:

  • ports=["PROTOCOL/PORT", "PROTOCOL/PORT",...] if you declare several ports.
  • port="PROTOCOL/PORT" if you declare only one port.

In this config file, we linked tcp/8022 and tcp/8023 to our service.

Secondly, we need to define the name of the service we want to link these ports to.
The convention is services=["<chosen_name_for_the_service>"].

More information about ports in concepts/services/#ports.

- [channel.console]

The channel defines by which way you want to deliver events, alerts and logs of HoneyTrap. You can define several channels.

[channel.console]
type="console"
For each channel:

Firstly, we need to define the name of the channel.
The convention is [channel.<you_choose_the_name>].
The name can be freely chosen and is used in the filter section.

Secondly, we need to define the type of the channel.
The convention is type=<official_name_of_channel>.

In this example, all the events will be logged in the console.

More information about channels in concepts/channels/.
More information about console in configuration/channels/console.
More configurations for channels in configuration/channels.

- [[filter]]

The filter selects the data that is send to a previously defined channel.

[[filter]]
channel=["console"]

We need to define on which channel this filter will be applied.
The convention is channel=["chosen_name_for_the_channel"]

We could have defined a type= to filter which events we want to pipe to this channel but in this example, for simplicity reasons, we didn’t do it. Therefore, each and every event will be logged into our chosen channel.

More information about filters in concepts/channels/#filters.

- [[logging]]

Logging defines the logging backend, needed to view log messages.

[[logging]]
output="stdout"
level="debug"

output defines the stream you want to use for logging.
Here, we chose to use stdout.

level defines the minimal level of events you want to log.
In this example, debug is the smallest level which means all the logs will appear.

More configurations for logging in configuration/logging/.
